# Roadmap разработки прототипа AI-агента адаптации сотрудников
## Для HORECA-франчайзинговой сети доставки суши и роллов

---

## Barista branch update · 2026-06-28

Ветка `codex/barista` временно меняет demo scope под запрос Valle Sanchez: отдельная версия приложения для входного тестирования и развития новых бариста.

Видимый MVP этой ветки:

| Модуль | Описание |
|---|---|
| Профиль бариста | Имя, грейд, кофейня, дата выхода |
| Demo база стандартов | Локальные mock-стандарты кофеварения с источниками и hashed-vector retrieval |
| Диагностика бариста | Вопросы по эспрессо, молоку, зерну, чистке, рецептурам, потоку и коммуникации |
| Skill gap analysis | Topic-level сильные зоны, зоны развития и критичные темы |
| План развития | Персональный маршрут day 1 / day 7 / day 14 |
| AI-наставник | Отвечает только по demo KB и показывает источники |
| Дашборд руководителя | Demo cohort + live localStorage results, риск, готовность, рекомендации и маршрут |

Ограничения этой ветки:

- demo-стандарты не являются финальными локальными правилами Valle Sanchez;
- external LLM, external embeddings, backend and production vector DB are not included;
- dashboard persistence is local to the browser;
- старые cook/admin данные могут оставаться в коде для стабильности тестов, но не являются видимым сценарием этой ветки.

## 1. Цель прототипа

Разработать демонстрационный прототип AI-агента, который помогает управляющей компании франчайзинговой сети стандартизировать и ускорить адаптацию новых сотрудников в точках.

В первой итерации прототип фокусируется на двух ролях:

1. **Повар**
2. **Администратор кафе / смены**

Ключевая механика:

> В первый день сотрудник проходит мягкую диагностику знаний на основе своей должности, грейда и базы знаний управляющей компании.  
> По результатам диагностики AI-агент формирует персональную траекторию обучения на 1, 7 и 14 дней.

---

## 2. Главная гипотеза прототипа

Если новый сотрудник в первый день проходит диагностику знаний, а затем получает персональный маршрут обучения, то компания:

- ускоряет выход сотрудника на смену;
- сокращает время обучения;
- не тратит время на темы, которые сотрудник уже знает;
- фокусирует обучение на слабых зонах;
- снижает стресс и негатив у новичка;
- уменьшает нагрузку на управляющего точки;
- повышает соблюдение стандартов сети.

## 5. Что не входит в первую итерацию

| Не включаем в MVP | Причина |
|---|---|
| Полная интеграция с HRM / LMS / кассой | Усложняет и удлиняет прототип |
| Автоматическая оценка качества реальной смены | Нужны операционные данные |
| Компьютерное зрение для кухни | Это отдельный технический блок |
| Полная библиотека всех ролей сети | В первой итерации достаточно повара и администратора |
| Предиктивная аналитика увольнений | Нужны исторические данные |
| Сложная система сертификации | Можно добавить после проверки гипотезы |

## Этап 4. Генерация диагностического теста

### Срок

**Неделя 3**

### Цель

Создать механику, при которой AI-агент формирует тест по должности, грейду и базе знаний.

### Логика работы

```text
Роль сотрудника
    ↓
Грейд сотрудника
    ↓
Карта компетенций
    ↓
База знаний УК
    ↓
AI-генерация вопросов
    ↓
Диагностика в первый день
```

### Формат диагностики

| Параметр | Значение |
|---|---|
| Длительность | 10–20 минут |
| Количество вопросов | 12–25 |
| Формат | Тесты, ситуационные вопросы, мини-кейсы |
| Тональность | Поддерживающая, не экзаменационная |
| Результат | Карта сильных и слабых зон |
| Повторная проверка | День 7 и день 14 |

### Пример вопроса для повара

```text
Ты заметил, что на контейнере с ингредиентом нет маркировки.
Что нужно сделать?

A. Использовать, если продукт выглядит нормально
B. Спросить старшего смены и не использовать до проверки
C. Поставить контейнер обратно
D. Использовать только в срочном заказе
```

Проверяемая тема: **маркировка и безопасность продукта**.

### Пример вопроса для администратора

```text
Гость звонит и говорит, что заказ задерживается на 25 минут.
Что ты сделаешь в первую очередь?

A. Скажешь, что кухня загружена
B. Проверишь статус заказа и сообщишь гостю конкретную информацию
C. Сразу предложишь отмену
D. Переведёшь звонок на повара
```

Проверяемая тема: **работа с претензией и статусом заказа**.

### Задачи

| Задача | Результат |
|---|---|
| Создать шаблоны вопросов | Тест, кейс, ситуационный вопрос |
| Настроить генерацию по роли | Повар получает кухонные вопросы, администратор — сервисные |
| Настроить генерацию по грейду | Новичку проще, опытному сотруднику сложнее |
| Связать вопрос с темой | Каждый вопрос проверяет конкретный блок |
| Связать вопрос с источником | Каждый вопрос основан на документе УК |
| Настроить мягкую подачу | Диагностика выглядит как персонализация обучения |

### Артефакты

- Генератор диагностики.
- Банк тестовых вопросов.
- Связка `вопрос → тема → документ → роль → вес`.
- Интерфейс прохождения диагностики.

### Критерии готовности

- Сотрудник может пройти диагностику.
- Вопросы отличаются для повара и администратора.
- Вопросы учитывают грейд.
- Каждый вопрос связан с темой и источником.

## Этап 5. Skill Gap Analysis

### Срок

**Неделя 4**

### Цель

На основе результатов диагностики определить сильные и слабые зоны сотрудника.

### Логика оценки

| Результат по теме | Интерпретация | Действие |
|---:|---|---|
| 80–100% | Тема понятна | Пропустить подробный блок или дать короткое резюме |
| 60–79% | Есть базовое понимание | Дать короткий модуль и 1–2 практических задания |
| 40–59% | Есть заметные пробелы | Назначить полноценный модуль и проверку |
| 0–39% | Критичная зона | Назначить обучение, наставника и повторную диагностику |
| Обязательная тема | Даже при хорошем результате | Дать краткий контрольный блок |

### Пример результата для повара

| Тема | Результат | Решение |
|---|---:|---|
| Гигиена и безопасность | 75% | Короткий обязательный блок |
| Техкарты | 48% | Полный модуль |
| Маркировка | 42% | Полный модуль + практика |
| Упаковка | 81% | Краткое резюме |
| Популярные позиции меню | 55% | Ускоренный модуль |
| Работа с браком | 38% | Обязательный модуль + проверка |

### Пример результата для администратора

| Тема | Результат | Решение |
|---|---:|---|
| Общение с гостем | 82% | Краткое резюме стандартов сети |
| Работа с заказом | 61% | Короткий модуль + практика |
| Кассовые операции | 45% | Полный обязательный модуль |
| Претензии | 39% | Полный модуль + кейсы |
| Стоп-лист | 52% | Ускоренный модуль |
| Закрытие смены | 30% | Назначить позже, не в первый день |

### Задачи

| Задача | Результат |
|---|---|
| Настроить scoring по темам | Оценка не общим баллом, а по блокам знаний |
| Определить сильные зоны | Что можно сократить |
| Определить слабые зоны | Что нужно усилить |
| Выделить критичные пробелы | Где нужен наставник или управляющий |
| Сформировать объяснение | Почему назначен именно такой маршрут |

### Артефакты

- Модель оценки по темам.
- Таблица результатов диагностики.
- Список сильных и слабых зон.
- Рекомендации по обучению.
- Сигналы риска для управляющего.

### Критерии готовности

- Система показывает не просто общий балл, а карту знаний.
- Для каждой слабой темы есть рекомендованное действие.
- Для обязательных тем действует отдельная логика.

## Этап 6. Построение персональной траектории обучения

### Срок

**Неделя 4–5**

### Цель

Сформировать индивидуальный маршрут обучения на 1, 7 и 14 дней.

### Логика маршрутизации

```text
Если тема > 80%:
- пропустить подробный модуль;
- дать короткое резюме;
- добавить контрольный вопрос.

Если тема 60–79%:
- дать короткий модуль;
- добавить 1–2 практических задания.

Если тема 40–59%:
- дать полный модуль;
- добавить повторную проверку.

Если тема < 40%:
- дать полный модуль;
- назначить помощь наставника;
- добавить повторную диагностику.

Если тема обязательная:
- не пропускать полностью;
- максимум сократить до короткого контрольного блока.
```

## Пример персонального маршрута: Повар

| Срок | Фокус | Задачи |
|---|---|---|
| День 1 | Ввод и безопасность | Диагностика, гигиена, рабочее место, правила кухни |

## 14. Риски и ограничения

| Риск | Что может пойти не так | Как закрываем |
|---|---|---|
| Диагностика воспринимается как экзамен | Сотрудник стрессует в первый день | Используем мягкую коммуникацию: “это нужно для персонального маршрута” |
| База знаний неструктурирована | AI даёт неточные рекомендации | Сначала размечаем темы, роли и обязательные блоки |
| Франчайзи не соблюдают процесс | Данные будут неполными | Дашборд УК показывает прохождение адаптации по точкам |
| Управляющий игнорирует рекомендации | Сотрудник не получает поддержку | Добавляем простые action items для управляющего |
| Сотрудник пропускает обучение | Маршрут не выполняется | Напоминания и прогресс в дашборде |
| AI ошибается в ответах | Риск неверного обучения | Ответы только на основе базы знаний, fallback на управляющего |
| Слишком широкий MVP | Прототип затягивается | Ограничиваемся 2 ролями и 14-дневной траекторией |

## 15. Definition of Done для прототипа

Прототип считается готовым, если:

- можно выбрать роль: повар или администратор;
- можно выбрать грейд сотрудника;
- система генерирует диагностику на основе роли и базы знаний;
- сотрудник проходит диагностику;
- система показывает сильные и слабые зоны;
- AI строит персональный маршрут на 1, 7 и 14 дней;
- сотрудник может задать вопрос AI-наставнику;
- управляющий видит прогресс, риски и рекомендации;
- УК видит агрегированные проблемные темы;
- в демо можно показать экономию времени обучения;
- в демо можно показать снижение стресса и повышение готовности к смене.
